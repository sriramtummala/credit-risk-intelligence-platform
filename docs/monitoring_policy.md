# Model Monitoring Policy

**Model:** Credit Risk Scoring — XGBoost Champion  
**Version:** 1.0  
**Status:** Draft — requires Credit Risk Committee approval before production use  
**Last updated:** 2026-06-30

---

## 1. Purpose and Scope

This document defines the operational monitoring standards for the Credit Risk Intelligence Platform. It establishes quantitative thresholds for data quality, feature drift, and model performance, and prescribes the escalation and incident-response procedures to follow when those thresholds are breached.

Monitoring is mandatory under **SR 11-7** (Federal Reserve / OCC Model Risk Management guidance):

> *"Ongoing monitoring... should include tracking of model performance... banks should have a process for identifying and responding to model deterioration."*  
> — SR 11-7, Section IV

A credit scoring model that silently degrades is a financial liability. Approving customers who are riskier than the model believes costs approximately **$5,000 per missed default**; the cost of an unnecessary decline is approximately **$500 in lost revenue**. Early detection of drift allows threshold adjustment or retraining before these costs accumulate.

---

## 2. Monitoring Framework

The platform monitors three distinct failure modes. Each requires different detection methods and different responses.

| Layer | What it detects | Primary metric | Report |
|-------|----------------|---------------|--------|
| Data Quality | Broken pipelines, upstream schema changes | Missing-value rate, null %, out-of-range counts | `reports/data_quality.html` |
| Feature Drift | Input distribution shift (population change) | Population Stability Index (PSI), KS statistic | `reports/data_drift.html` |
| Model Performance | AUC / recall degradation | AUC-ROC, Gini, approval rate | `reports/model_performance.html` |

Evidently AI reports are generated by running `notebooks/09_evidently_monitoring.ipynb` against the current production batch. The reference dataset is the original 30,000-row UCI training population.

## 3. Data Quality Thresholds

Data quality checks run on every production batch before the model is invoked. A failed data quality check is a **pipeline problem**, not a model problem — the root cause is an upstream data feed or ETL change.

### 3.1 Missing Value Limits

| Feature | Alert threshold | Critical threshold |
|---------|----------------|--------------------|
| `LIMIT_BAL` | > 1% missing | > 5% missing |
| `PAY_0` | > 0.5% missing | > 2% missing |
| `PAY_AMT1` – `PAY_AMT6` | > 1% missing | > 5% missing |
| `BILL_AMT1` – `BILL_AMT6` | > 1% missing | > 5% missing |
| `AGE` | > 0.1% missing | > 1% missing |
| All other features | > 2% missing | > 10% missing |

`PAY_0` has a tighter threshold because it is the **top SHAP driver** (mean |SHAP| = 0.389); missing values in this feature degrade model accuracy disproportionately.

### 3.2 Range Violation Limits

| Feature | Valid range | Action if violated |
|---------|------------|-------------------|
| `AGE` | 18 – 100 | Flag record; do not score |
| `LIMIT_BAL` | > 0 | Flag record; do not score |
| `PAY_0` – `PAY_6` | −2 to 8 | Alert data engineering |
| `SEX` | {1, 2} | Alert data engineering |
| `EDUCATION` | {0, 1, 2, 3, 4, 5, 6} | Bin to "Other"; log anomaly |
| `MARRIAGE` | {0, 1, 2, 3} | Bin to "Other"; log anomaly |

### 3.3 Batch-Level Sanity Checks

| Check | Alert threshold | Critical threshold |
|-------|----------------|--------------------|
| Batch row count vs. expected | < 80% or > 120% of expected | < 50% or > 200% |
| Duplicate record rate | > 0.5% | > 2% |
| Schema column count | Any missing column | — |

**Response to a data quality alert:** Do not suppress scoring — score available records and flag the anomaly in the audit log. Notify data engineering immediately. If a critical threshold is breached, halt batch scoring and escalate to the Model Owner.

## 4. Feature Drift Thresholds (PSI)

The Population Stability Index measures how much the distribution of a feature has shifted between the reference population and the current production batch.

$$\text{PSI} = \sum_{i=1}^{n} (A_i - E_i) \ln\!\left(\frac{A_i}{E_i}\right)$$

where $A_i$ = actual (production) proportion in bin $i$, $E_i$ = expected (reference) proportion.

### 4.1 PSI Severity Scale

| PSI range | Status | Colour | Required action |
|-----------|--------|--------|----------------|
| < 0.10 | Stable | Green | Continue monitoring; no action |
| 0.10 – 0.25 | Moderate shift | Yellow | Investigate the feature; check upstream data pipelines; monitor weekly |
| > 0.25 | Major shift | Red | Alert Model Owner; initiate expedited retraining review |

### 4.2 Per-Feature PSI Thresholds

Features are monitored individually. A single feature breaching the Red threshold is sufficient to trigger escalation.

| Feature | Monitoring frequency | Notes |
|---------|---------------------|-------|
| `LIMIT_BAL` | Weekly | Economic indicator; sensitive to rate cycles |
| `AGE` | Monthly | Demographic; slow-moving |
| `PAY_0` | Weekly | Top SHAP driver — highest priority |
| `PAY_2` – `PAY_6` | Weekly | Secondary delinquency signals |
| `BILL_AMT1` – `BILL_AMT6` | Weekly | Correlated with LIMIT_BAL; monitor together |
| `PAY_AMT1` – `PAY_AMT6` | Monthly | Payment behaviour; slower to shift |

### 4.3 KS Statistic

In addition to PSI, a two-sample Kolmogorov-Smirnov (KS) test is run per numerical feature. A p-value < 0.05 flags a statistically significant distributional shift. KS results inform the severity assessment alongside PSI — PSI alone can be misleading for small batches.

## 5. Model Performance Thresholds

### 5.1 Primary Performance Metrics

| Metric | Baseline (training) | Alert | Critical | Action |
|--------|-------------------|-------|----------|--------|
| AUC-ROC | 0.7725 | < 0.72 | < 0.70 | See §7 |
| Gini coefficient | 0.5450 | < 0.45 | < 0.40 | See §7 |
| Recall at t=0.30 | 80.6% | < 72% | < 65% | See §7 |
| Approval rate | ~70% (baseline) | ± 10 pp | ± 20 pp | Business review |

### 5.2 Sustained Degradation Rule

A single below-threshold reading may reflect a noisy batch rather than true model deterioration. The following sustained-degradation rules apply:

| Condition | Response |
|-----------|----------|
| AUC-ROC < 0.72 for **one** consecutive batch | Alert Model Owner; expedited monitoring |
| AUC-ROC < 0.72 for **two** consecutive weeks | Mandatory retraining initiation |
| AUC-ROC < 0.70 for **any** batch | Immediate escalation; consider offline review |
| Approval rate deviation ± 10 pp for **two** consecutive weeks | Business and compliance review |

### 5.3 Approval Rate Monitoring

The approval rate (fraction of applicants scored below threshold t=0.30) is a leading indicator of score-distribution shift. It does not require ground-truth labels and can be computed in real time.

| Condition | Meaning | Response |
|-----------|---------|----------|
| Rate rises > 10 pp above baseline | Model is approving more risk than expected | Threshold review; tighten to t=0.25 temporarily |
| Rate falls > 10 pp below baseline | Model is over-declining | Threshold review; check feature values |
| Rate shifts > 20 pp in either direction | Severe score distribution shift | Escalate to Credit Risk Committee immediately |

## 6. Concept Drift Detection

Concept drift — where the *relationship* between features and the target changes — is the hardest failure mode to detect because input distributions may look normal while the model's predictions become invalid.

### 6.1 Detection Methods

| Signal | How it is computed | Alert condition |
|--------|-------------------|----------------|
| `corr(PAY_0, default)` rolling correlation | 30-day rolling Pearson correlation on labelled outcomes | Correlation drops below +0.20 (reference ≈ +0.33) |
| SHAP rank stability | Compare top-5 SHAP feature rank order vs. reference | Any rank inversion in top-3 features |
| AUC on labelled holdout | Scored on a monthly holdout with realised labels | AUC < 0.72 (same as §5.1) |

### 6.2 Why Concept Drift Requires Different Handling

PSI on `PAY_0` may remain low (the feature values look normal) while its predictive relationship with default collapses. The only reliable detection is **outcome monitoring** — tracking realised default rates in the approved and declined populations against model predictions.

In a regulated environment (SR 11-7), outcome analysis must be conducted at a minimum **quarterly**, with results documented and retained for model audit review.

## 7. Incident Response Plan

When any Alert or Critical threshold is breached, the following procedure applies. All steps must be documented in the model risk log.

### Step 1 — Alert the Model Owner

**Trigger:** Any threshold listed in §3, §4, or §5 is breached.

- The monitoring system generates an automated alert (Evidently report + email/Slack notification) within 4 hours of the batch run.
- The Model Owner acknowledges the alert within **1 business day**.
- If the Model Owner is unavailable, escalation goes to the Credit Risk Committee Chair.

### Step 2 — Triage: Data Issue vs. Market Change

Before initiating retraining, determine the root cause:

| Investigation question | Data issue | Market change |
|-----------------------|-----------|---------------|
| Did upstream data pipelines change schema? | Yes → Data engineering fix | No |
| Is drift isolated to one feature? | Possibly data issue | No |
| Is the default rate itself shifting? | Unlikely | Yes — economic event |
| Are all features drifting simultaneously? | Possible ETL failure | Yes — macro shift |

**Data issues** are resolved by fixing the pipeline and re-scoring. Retraining is not required.  
**Market changes** require retraining with data from the new economic period.

### Step 3 — Threshold Adjustment (If Appropriate)

If model performance has degraded due to a shift in the default rate (target drift), the classification threshold may be adjusted before retraining is complete:

- Increase t (more conservative) if the default rate has risen — reduces approval rate to match new risk level.
- Decrease t (less conservative) if the default rate has fallen — avoids over-declining creditworthy applicants.

Threshold changes must be documented in `docs/threshold_policy.md` (revision history) and communicated to the Credit Risk Committee within **2 business days**.

### Step 4 — Trigger the Retraining Pipeline

If retraining is warranted:

```bash
# Retrain on updated data (replace with current data path)
python main.py --download   # or point to updated dataset

# Review new model metrics before promoting
```

The new candidate model must meet or exceed the following minimum bar before promotion:

| Metric | Minimum to promote |
|--------|--------------------|
| AUC-ROC on new holdout | ≥ 0.74 |
| Recall at t=0.30 | ≥ 75% |
| Gini coefficient | ≥ 0.48 |

### Step 5 — Champion vs. Challenger Comparison

Before deploying the retrained model, run a side-by-side evaluation:

| Comparison dimension | Tool |
|---------------------|------|
| AUC, Gini, Recall | `src/evaluate.py` on shared holdout |
| SHAP feature importance rank | `notebooks/07_shap_explainability.ipynb` |
| Score distribution | PSI between champion and challenger scores |
| Business cost | Expected cost at t=0.30 (FN=$5K, FP=$500) |

The challenger must outperform the champion on **at least 3 of 4 dimensions** to be promoted. The comparison results are appended to `docs/release_notes_v1.md` under a new version entry.

### Step 6 — Deployment and Documentation

After promotion:

1. Replace `models/champion_model.joblib` and `models/champion_model_metadata.json`.
2. Update the version field in `champion_model_metadata.json`.
3. Restart the API service (the lifespan handler loads new artifacts on startup).
4. Append an entry to the revision history in this document and in `docs/release_notes_v1.md`.
5. Notify the Credit Risk Committee that the model has been updated.

## 8. Roles and Responsibilities

| Role | Responsibility |
|------|---------------|
| **Model Owner** (Credit Risk Analytics) | Receives alerts; approves retraining; signs off on champion promotion |
| **Data Engineering** | Investigates data pipeline issues; fixes upstream feed problems |
| **Credit Risk Committee** | Reviews threshold changes; approves model changes with material business impact |
| **Model Risk Management** | Independent validation of challenger before promotion (SR 11-7 requirement) |
| **Technology / MLOps** | Executes deployment; maintains monitoring infrastructure |
